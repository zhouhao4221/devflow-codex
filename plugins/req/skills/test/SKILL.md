# 需求测试

针对指定需求执行综合测试：运行已有测试 → 引导创建新测试 → 交互验证测试要点。

> 测试文件位置、运行命令、环境启动从 `docs/prompt/testing.md` 读取。文件不存在时打印创建提示，回退到 `docs/prompt/architecture.md` 测试规范章节。

## 命令格式

```
/req:test [REQ-XXX] [选项]
```

省略编号时自动选择「开发中/测试中」的需求，多个候选让用户选择。

| 选项 | 说明 |
|-----|------|
| `--failed` | 仅运行上次失败的测试 |
| `--skip-ut` / `--skip-api` / `--skip-e2e` | 跳过对应阶段 |
| `--force` | 某阶段失败时继续后续 |

---

## 总体流程

1. 选择需求 & 前置检查
2. 提取测试要点（业务维度）
3. 识别变更范围（代码维度）
4. 阶段一：UT（变更相关的已有 + 缺失引导创建）
5. 阶段二：API 测试（已有 + 缺失引导创建）
6. 阶段三：E2E 测试（已有 + 缺失引导创建）
7. 交互验证：自动化未覆盖的测试要点
8. 更新状态、记录结果、同步缓存
9. 汇总报告

---

## 详细流程

### 1. 选择需求 & 前置检查

指定编号或自动查找。状态必须为「开发中/测试中」。功能点有未完成时警告。

### 2. 提取测试要点

从需求文档提取，作为验证基准。分四类：API 测试、业务规则测试、数据权限测试、其他测试。

### 3. 识别变更范围

优先从需求文档「文件改动清单」提取，否则 `git diff` 对比当前分支与主分支。按 `docs/prompt/testing.md` 定位关联测试文件。

---

### 阶段一：UT（变更相关）

**回归**：运行变更相关的已有 UT（命令从 testing.md 读取），将通过的用例匹配到测试要点。

**新建**：对缺少 UT 的变更文件，引导执行 `/req:test-new --type=ut`。

---

### 阶段二：API 测试

**环境检查**：按 testing.md 检查依赖服务和后端服务，未启动时自动启动并等待就绪。

**回归**：运行已有 API 测试，匹配测试要点。

**新建**：对未覆盖的 API 测试要点，引导执行 `/req:test-new --type=api`。

---

### 阶段三：E2E 测试

**环境检查**：额外检查前端服务。

**回归**：运行已有 E2E 测试。

**新建**：对未覆盖的用户流程，引导执行 `/req:test-new --type=e2e`。

---

### 交互验证

对自动化未覆盖的测试要点，逐项引导用户手动验证（展示操作步骤、预期结果），记录通过/失败/跳过。

---

### 更新状态 & 汇总报告

- 状态改为「测试中」，更新生命周期
- 勾选测试要点通过项，记录日期和覆盖情况
- 更新主仓需求文档；legacy 缓存同步仅在旧项目需要时执行
- 输出汇总报告（各阶段通过/失败、测试要点覆盖率）

**全部通过**：提示 `/req:done`。**存在失败**：列出失败用例和原因，提示 `/req:dev` 修复或 `--failed` 重跑。

---

## 测试模式

| 模式 | 命令 |
|------|------|
| 综合测试（默认） | `/req:test REQ-XXX` |
| 增量测试 | `/req:test REQ-XXX --failed` |
| 跳过阶段 | `--skip-ut` / `--skip-api` / `--skip-e2e` |
| 强制继续 | `--force` |
